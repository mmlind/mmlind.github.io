How to use multiple cores to process very large text files

Oct 8, 2020 Natural Language Processing (NLP) Multiprocessing C Python
Share on:

If you work on natural language processing (NLP) one of the first things you often want to do is to get a dictionary of unique words from a given text corpus. I recently did this with one of the larger modern text corpora that exists: the Wikipedia text. The English Wikipedia text spans approximately 2.5 billion words in a 14 Gb text file. Given the size of this corpus, I wanted to assess to what extent multiple processors could be utilized to fasten the process.

In this blog post I introduce 5 technical methods that utilize multiple processes on a modern multi-core computer to count unique words, or tokens, in the Wikipedia text corpus. In particular, I explore how scalable these techniques are by simply adding more cores.

Reading large Wikipedia files

A frequent source of text corpora for NLP practitioners is Wikipedia. Regular backups are available at dumps.wikimedia.org, the latest dumps of the English site can be downloaded here. Simply look for a file enwiki-latest-pages-articles.xml.bz2 which includes all current articles.

Naturally, these files are not small. The current version of the English Wikipedia dump file is a whopping 17.79 GB, compressed. One of the reasons for the enourmous filesize is that all dump files come in a clumsy XML format which carries a lot of repetitive, unneeded baggage.

For this excercise I download the version dated Oct. 2nd 2020. Since I’m interested in a plain text corpus, I need to strip off the XML markup and all the meta data that is included. Fortunately, there are a number of tools out there that can help with task, for example: Building a Wikipedia Text Corpus for Natural Language Processing, Word2vec 4 Wikipedia or WP2TXT.

Creating a dictionary from a corpus

After conversion to plain text the dump file of the English Wikipedia is still about 14 GB. This is the file that I am going to work on. The objective is to read through this file as fast as possible, to create a dictionary of all its unique words, and to sort the words by their count (i.e. how many times they appear in the corpus).

The benchmarks

I want to investigate to what extent processing the corpus with multiple cores, either using multiprocessing or multithreading, can speed up the process. Before I start assessing the different technical options for parallel processing I first want to define the 2 benchmarks to compare against:

  1. processing via Python, and
  2. processing via C using a single process.

So, let’s first explore the performance of creating the dictionary with Python.

Sequential processing with C

My second benchmark shall be creating the dictionary with a simple, sequential C program, running on a single core. In this section I will explain the main steps how this can be done. Doing so also allows me to introduce some of the code that will be used later in the parallel processing techniques as well.

If we break down the algorithm, creating the dictionary involves 5 main steps:

  1. Map the corpus file to memory
  2. Read the corpus word by word
  3. Insert each word into a hash table
  4. Copy the words and their count into an array
  5. Sort the array by count in descending order

While this is clearly not as efficient and elegant as doing so in Python, let’s see what performance improvement we may be able to achieve, even using a single core.

Map the corpus file to memory

Wikipedia dump files can be huge. To ensure they can be read even on computers that do not have sufficient memory to load the complete file at once, I use mmap to map the physical file into memory. The loading into memory is done smartly based on need (ie. what part of the file is being read), and the operating systems helps caching pages which further improves performance.

Using mmap is simple and straight forward. I create below function map_shared_file() to do and hide all the usual error handling. The function returns a pointer to the mapped file which allows to read the file simply by accessing this pointer. The function also returns the file size of the corpus which we will need later.

 1void *map_shared_file(const char *fname, size_t *fsize){
 2        
 3    struct stat fstat;
 4
 5    if (stat(fname, &fstat) == -1) {perror("[STAT] Error getting file stats! ABORT. ");exit(1);}
 6
 7    *fsize = fstat.st_size;
 8    
 9    int fd = open(fname, O_RDONLY);
10    if (fd == -1) {perror("[OPEN] Error opening file for shared use! ABORT. ");exit(1);}
11    
12    char *mem = mmap((void*)0, *fsize, PROT_READ, MAP_SHARED, fd, 0);
13    
14    if (mem == (void*)(-1)) {perror("[MMAP] Error creating shared memory for file reading! ABORT. ");exit(1);}
15    
16    if (close(fd)==-1) {perror("[CLOSE] Error closing mapped file! ABORT. ");exit(1);}
17
18    return mem;
19
20}

BTW: I also tested fully loading the corpus into memory which, however, surprisingly did not result in any measurable performance improvement. To the contrary, in some scenarios mapping the file was even slightly faster than fully loading the file into memory.

Read the corpus word by word

Before we now move on to read the corpus word by word, I want to introduce a basic structure for representing a ‘word’ that I will use thoughout the program. I call it the corpus_token_t:

1// basic structure to represent a 'word/token' that exists in the corpus
2typedef struct corpus_token_t{
3    size_t off;                 // offset of 1st occurance in corpus
4    int len;                    // length of the token, not null-terminated
5    size_t count;               // counter for how many times this token is found
6} corpus_token_t;

As you noticed, from now on we refer to tokens instead of words because in some languages, for example in Chinese, the most basic element of a sentence or a text is not a word but a character. A token may thus be either a series of characters, i.e. a word, or a single character.

Beware that the corpus_token_t structure does not include a string of the actual character representation of the respective token. Instead, I only store the offset of the first occurance of this token in the corpus; plus its length and a counter for how many times this token exists in the corpus. The string is actually not needed and avoided for faster performance.

Reading the corpus word by word is a bit trickier than it may seem at first. Why? There are 2 things to consider: multi-byte sequences and word boundaries.

Multi-byte sequences

I want to be able to read Wikipedia dumps (or any other text files for that matter) of any language. Therefore, I can’t read the file simply char by char because in many languages a single character uses more than a single byte char.

UTF8 has become a de facto standard. Unfortunately, in UTF8 there is no fixed character length. Characters may range in size from 1 to 4 bytes which makes handling UTF8 in C a bit of a pain.

Fortunately, there are some small libraries to come to rescue, for example Jeff Bezanson’s excellent ground work for UTF8 processing called ‘cutef8’. I use his simple macro for identifying whether a char is UTF8, …

1#define IS_UTF8(c) (((c)&0xC0)!=0x80)

… and expand on his original u8_memchr function to create the following get_next_token function which reads a text segment of given length token by token:

 1corpus_token_t get_next_token(char *corpus, size_t *idx, size_t end){
 2
 3    corpus_token_t tok;
 4    size_t start_idx = *idx;
 5
 6    // move IDX forward in case it points INSIDE of a multibyte character
 7    while (!IS_UTF8(corpus[*idx]) && (*idx < end) && corpus[*idx]) (*idx)++;
 8    
 9    // move IDX gradually forward
10    while (*idx<end && *idx - start_idx<MAX_TOKEN_SIZE && corpus[*idx] ){
11        
12        uint32_t u8_char_code = 0;
13        int      u8_char_size = 0;
14        
15        do {
16            u8_char_code <<= 6;
17            u8_char_code += (unsigned char)corpus[*idx];
18            u8_char_size++;
19            (*idx)++;
20        } while (*idx<end && corpus[*idx] && !IS_UTF8(corpus[*idx]));
21
22        u8_char_code -= UTF8_OFFSETS[u8_char_size-1];
23
24        // 3 exit conditions: (1) multi-byte character, (2) end of text, (3) token delimiter
25        bool is_del = is_token_delimiter(u8_char_code); // this fct will be introduced below
26        if (u8_char_size>1  || *idx==end || is_del) {
27            tok.len   = is_del ? (int)(*idx - start_idx - u8_char_size) : (int)(*idx - start_idx);
28            tok.off   = start_idx;
29            tok.count = 1;
30            return tok;
31        }
32        
33    }
34    
35    // if no token was found, return empty token reference
36    tok.len   = 0;
37    tok.off   = 0;
38    tok.count = 0;
39    
40    return tok;
41}

In most Western languages the obvious answer is a whitespace between two characters. But of course there are many other characters. Insert each word into a hashtable

When we process the corpus we need to deduplicate all tokens to get a list of unique tokens only. One of the most efficient tools for deduplication and quick lookups is a hashtable. If you are not familiar with hashtables I suggest to catchup with some respective reading.

In our case we want to use the string of the token as hash key. To implement the hashtable we need 4 components:

  1. Creating the hashtable
  2. Hashing a string
  3. Comparing tokens
  4. Inserting a token into the hashtable
Creating the hashtable

The hashtable itself is nothing but a an array of corpus_token_t structs. That’s easy. But what about its size? There is no agreed definition of the ideal or best size of a hashtable. Some argue that a hashtable should be big enough that the number of expected entries in the hashtable will not occupy more than 70% of its space.

Since I want the size of the hashtable be somewhat relative to the size of the corpus I use the following formula to define the hash size:

1MAX_HASH_SIZE = 100000 * pow(2,log10(corpus_fsize));
2
3corpus_token_t *tok_htab = calloc(MAX_HASH_SIZE, sizeof(corpus_token_t));

You can and should try different sizes. Later on, during multiprocessing and multithreading, we will find that the size of the hashtable may considerably impact the performance of the overall algorithm. If you use fixed sizes, using prime numbers arguably makes better hashes.

I used this online tool to quickly find prime numbers of any size. Below are some of the alternatives that I used.

Hashing a string

Next, we want to hash each token to obtain a unique code. I use Daniel J. Bernstein’s DJB2 hash function which is simple and works well for strings. Feel free to try and compare with others.

Remember that our corpus_token_t structure does not include the actual character representation (aka string) of this token.

 1void get_token_string(corpus_token_t tok, char *str, char *corpus){
 2    
 3    memcpy(str, corpus + tok.off, tok.len);
 4    memset(str + tok.len, '\0', 1);
 5
 6    // convert to lower case
 7    char *p = str;
 8    while ((*p = tolower(*p))) ++p;
 9
10    return;
11}

Therefore, we first need to create the actual string for this token before we can create a hash. I also convert to all tokens to lowercase to more efficiently use the hashtable and because for most NLP tasks the case of a word is insignificant.

 1size_t get_token_hash(corpus_token_t tok, char *corpus){
 2    
 3    // create a character representation of the token
 4    char tok_str[MAX_TOKEN_SIZE+1];
 5    get_token_string(tok, tok_str, corpus);
 6    
 7    // hash the token string
 8    
 9    char *str = tok_str;
10    
11    size_t hash = 5381;
12    
13    int c;
14
15    while ((c = *str++)) hash = ((hash << 5) + hash) ^ c; /* hash * 33 XOR c */
16
17    return hash % MAX_HASH_SIZE;
18        
19}
Comparing tokens

When we insert a token into a hash, we need to check for potential collisions. Therefore, we need a comparison function to assess whether two tokens are equal:

 1// compare two integers and return the smaller one
 2#define MIN_OF_TWO(a,b)         \
 3  ({ __typeof__ (a) _a = (a);   \
 4      __typeof__ (b) _b = (b);  \
 5    _a < _b ? _a : _b; })
 6
 7
 8int compare_corpus_tokens(const corpus_token_t *tok1, const corpus_token_t *tok2, const char *corpus){
 9    
10    int len1 = (*tok1).len;
11    int len2 = (*tok2).len;
12
13    int len = MIN_OF_TWO(len1, len2);
14    
15    const char *str1 = corpus + (*tok1).off;
16    const char *str2 = corpus + (*tok2).off;
17
18    int r = strncasecmp(str1, str2, len);
19    
20    // both strings differ in the first 'len' chars
21    if (r!=0) return r;
22    
23    // both strings are the same
24    if (len1 == len2) return 0;
25    
26    // if both strings are different length but share the first 'len' characters
27    // then return a reference to the shorter one
28    
29    if (len1 < len2) return -1;
30    
31    return 1;
32}
Inserting a token into the hashtable

Next, we insert each token into the hashtable. If the token exists already, its counter will be incremented:

 1void insert_token(corpus_token_t tok, corpus_token_t *htab, char *corpus){
 2    
 3    size_t hidx = get_token_hash(tok, corpus);
 4    
 5    while(1) {
 6        // check if this bucket is empty
 7        if (htab[hidx].len >0) {
 8            
 9            // if the same token already exists in the hash table increment its counter
10            if (compare_corpus_tokens(&tok, htab + hidx, corpus) == 0) {
11                htab[hidx].count += tok.count;
12                return;
13                
14            }
15            // otherwise move forward to the next bucket
16            else {
17                hidx++;
18                hidx %= MAX_HASH_SIZE;
19            }
20            
21        } else {
22            // if this token's bucket was empty (i.e. first time) add it into the hashtable
23            htab[hidx] = tok;
24            return;
25        }
26        
27    }
28    
29}

At the end, the number of filled entries or buckets of the hashtable gives us the number of unique tokens in the dictionary.

Copy the words and their count into an array

After the hashtable has been filled, the fourth step in our overall algorithm is copying all unique tokens into a more compact representation, into an array of tokens, the dictionary.

Remember the hashtable is sparsely filled and therefore ‘wastes’ a lot of memory. It also can’t be sorted which is why we need the copying into this new, more compact representataion.

 1corpus_token_t *create_dictionary(corpus_token_t *htab, char *corpus, size_t dict_tok_count, size_t *corpus_tok_count){
 2    
 3    *corpus_tok_count = 0;
 4
 5    corpus_token_t *dict = calloc(dict_tok_count, sizeof(corpus_token_t));
 6
 7    size_t didx = 0;
 8    for (size_t hidx=0; hidx<MAX_HASH_SIZE; hidx++){
 9    
10        if (htab[hidx].len>0) {
11            *corpus_tok_count += htab[hidx].count;  // calc overall token count
12            dict[didx++] = htab[hidx];              // add token into dictionary
13        }
14        
15    }
16        
17    assert(didx == dict_tok_count);
18    
19    return dict;
20}

Sort the array in descending order

The last step is simply sorting the dictionary array by the token counter, in descending order.

 1int compare_token_counters(const void *a, const void *b){
 2    
 3    size_t freq_a = (*(corpus_token_t*)a).count;
 4    size_t freq_b = (*(corpus_token_t*)b).count;
 5
 6    if (freq_a < freq_b) return 1;
 7    if (freq_a > freq_b) return -1;
 8    return 0;
 9}
10
11
12void sort_dict(corpus_token_t *dict, size_t dict_tok_count){
13    
14    qsort(dict, dict_tok_count, sizeof(corpus_token_t), compare_token_counters);
15
16}
